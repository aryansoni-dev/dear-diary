# Android Exported Component Source Audit

Audit date: 2026-07-17

## Scope and result

This audit traced the MobSF findings against a fresh Expo Android prebuild and the Gradle `releaseRuntimeClasspath` used by the Preview APK. It did not change `eas.json`, generated Android sources, dependencies, manifests, or build behavior.

The four exported activities are real release-manifest entries. They are not introduced by DearDiary application code:

- `androidx.compose.ui.tooling.PreviewActivity` comes from the manifest embedded in `androidx.compose.ui:ui-tooling-android:1.11.2`. The release dependency has three relevant owners: Clerk Android UI, Expo Dev Launcher, and Expo Dev Menu.
- The three `InstrumentationActivityInvoker` activities come from the manifest embedded in `androidx.test:core:1.7.0`. The only release path is Clerk: `@clerk/expo` -> `com.clerk:clerk-android-ui:1.0.23` -> `androidx.test:core-ktx:1.7.0` -> `androidx.test:core:1.7.0`.
- `android.permission.REORDER_TASKS` comes from the same AndroidX Test Core manifest and is not needed by DearDiary runtime behavior.
- `exp+dear-diary://` is generated by the installed `expo-dev-client` config plugin from the Expo slug. It is expected while that plugin runs, but it is not needed by a standalone Preview or Production build.

## Does Preview contain Expo Dev Client?

Yes, at the native dependency/artifact level.

- `expo-dev-client` is a direct application dependency.
- Effective Expo config lists `expo-dev-client`, `expo-dev-launcher`, and `expo-dev-menu` in plugin history and autolinked modules.
- Gradle applies `expo-dev-launcher-gradle-plugin` and includes all three projects on `releaseRuntimeClasspath`.
- Their release native code and resources are compiled. Release implementations are disabled stubs: Dev Launcher reports that it is unavailable, and Dev Menu is a disabled implementation.
- The debug-only dev-launcher activities and `expo-dev-launcher://` authentication filter are not merged into the release manifest. The only dev-launcher manifest contribution found in release is the ML Kit barcode metadata.

Therefore `developmentClient: false` prevents a functional development-client build, but it does not uninstall or stop autolinking an `expo-dev-client` package that is present in `package.json`.

## Why the AndroidX test activities are present

`com.clerk:clerk-android-ui:1.0.23` publishes `androidx.test:core-ktx:1.7.0` as a runtime dependency. Core KTX then pulls `androidx.test:core:1.7.0`. Because this is on `releaseRuntimeClasspath`, Android's normal library-manifest merge imports Test Core's permission, themes, and three activities into the application release manifest.

This is dependency metadata leakage from a production UI dependency, not an Android instrumentation-test APK being combined with the app. Expo's own AndroidX Test declarations are correctly scoped as `testImplementation` and do not enter the release graph.

## External launchability

All four MobSF activities have `android:exported="true"` and no component permission. Another installed app or ADB can address each one with an explicit component intent.

- `PreviewActivity` has no intent filter, so it is not discoverable as a normal launcher entry, but explicit launch is allowed.
- The AndroidX Test activities contain a low-priority `LAUNCHER` category but no `MAIN` action. They should not appear as normal launcher icons, but explicit launch is still allowed.

This audit did not launch the activities on a device. The conclusion is based on the final release manifest's Android access controls, not on an assumption about whether each activity performs useful work after launch.

## `REORDER_TASKS`

`android.permission.REORDER_TASKS` is declared by AndroidX Test Core immediately before its `InstrumentationActivityInvoker` activities. DearDiary does not declare or use it directly, and no other merged-manifest source contributed it. It supports the test activity-invocation machinery and is unnecessary in Preview and Production once AndroidX Test Core is removed from the release runtime graph.

The permission should not be removed on its own. Removing the source dependency is the smaller and more maintainable correction because it also removes the three unused exported activities and their test-only themes.

## `exp+dear-diary://`

The scheme is deterministic and expected with the current dependency set. The `expo-dev-client` plugin converts the configured slug `dear-diary` to `exp+dear-diary` and appends it to the exported `MainActivity` deep-link filter.

It is distinct from `app.json`'s intentional `deardiary://` application scheme. The latter supports DearDiary routing/auth callbacks and must be preserved. The generated `exp+dear-diary://` scheme is for launching a development client and is not required in a standalone Preview APK or Production AAB.

## Preview and Production comparison

Both profiles build the Android `release` variant. The repository has static `app.json` configuration and no profile-conditional `app.config.*`. The Preview/Production differences in `eas.json` are environment selection and APK versus app-bundle output; neither changes the dependency declarations or release manifest merge examined here.

Consequently, without a dependency change, the same four activities, `REORDER_TASKS`, generated `exp+dear-diary://` scheme, and dev package release artifacts are expected in Production as well.

## Recommended minimal fix

Do not add activity-by-activity or wildcard manifest suppression.

The smallest dependency-level correction that removes the MobSF findings is:

1. Keep Clerk, but stop `androidx.test:core-ktx`, `androidx.test:core`, and `androidx.compose.ui:ui-tooling` from entering non-test release runtime resolution. Prefer a compatible upstream Clerk/Clerk Android UI release with corrected dependency scopes. If no verified compatible release exists, use a small local Expo config plugin to generate narrowly targeted Gradle exclusions, then prove Clerk sign-in, profile, SSO, and release compilation still work.
2. Separately keep `expo-dev-client` available to the Development profile while omitting `expo-dev-client`, `expo-dev-launcher`, and `expo-dev-menu` from Preview/Production native autolinking and config-plugin execution. That removes the redundant Expo UI-tooling paths, dev resources/metadata, and `exp+dear-diary://` without changing the valid `eas.json` profiles.
3. Regenerate in a clean temporary directory and require the merged release manifest and release runtime graph to contain none of the four activities, no AndroidX Test Core, no `REORDER_TASKS`, no dev-launcher/menu component, and no `exp+dear-diary` scheme. Preserve `deardiary://` and all required Clerk, RevenueCat, Expo Notifications, and normal deep-link components.

No fix was implemented as part of this audit.
